# STG 系统技术实现详解

本文档用于解释当前实现的整体流程与关键机制，重点覆盖：

- 系统端到端处理路径
- 节点更新策略
- LLM 判定接入与回退
- 边冲突处理与时态失效
- attached owner 判定与切换
- 结果产物与调试入口

## 1. 模块总览

核心代码位于 `stg_system` 目录。

- `config.py`: 运行参数与阈值配置
- `geometry.py`: 几何计算
- `models.py`: 节点/边/图状态数据模型
- `candidate_resolver.py`: `get_candidates` 候选检索
- `llm_decider.py`: LLM 判定封装
- `node_processor.py`: 节点与边的核心处理逻辑
- `pipeline.py`: 端到端离线流水线编排
- `storage.py`: Neo4j 持久化
- `evaluation.py`: 验收报告构建
- `cli.py`: 命令行入口

## 2. 端到端流程

入口命令是：

```powershell
python -m stg_system.cli --input data/less_move.json --max-frames 20
```

执行顺序：

1. `cli.py` 解析参数，构建 `EngineConfig`
2. `SpatialTemporalPipeline.run()` 读取 JSON 帧数据
3. 对每一帧调用 `FrameProcessor.process_frame()`
4. 每帧先处理节点，再更新动态生命周期，再处理边
5. 若启用 Neo4j，批量 upsert 节点与边
6. 输出：
   - `semantic_graph_snapshot.json`
   - `design_acceptance_report.json`

## 3. 节点处理流程（node_process）

节点处理在 `FrameProcessor.process_frame()` 内完成。

### 3.1 候选检索

对每个对象 idx 调用 `get_candidates()`，返回三类候选：

- `cur_cmp`: 本帧高 IoU 重复候选（去重主集合）
- `cur_context`: 本帧上下文候选（辅助判定）
- `pre`: 图中历史节点候选（跨帧匹配）

### 3.2 去重与匹配（node_resolve）

`_node_resolve()` 是核心：

1. 决定 `primary_idx`
   - 优先 LLM：`decide_frame_dedupe`
   - 回退规则：取最高 score
2. 决定实体类型 `entity_type`
   - 规则初判（layer_id + moving_labels）
   - 可被 LLM 结果覆盖
3. 跨帧匹配历史节点
   - 优先 LLM：`decide_node_match`
   - 回退规则：标签优先 + IoU 几何检查
4. 新建或更新节点
   - 无匹配：`_create_node`
   - 有匹配：`_update_node`

### 3.3 反思更新（candidate）

更新阶段会走三种反思：

- `_handle_type_reflection`
- `_handle_label_reflection`
- `_handle_attribute_reflection`

策略要点：

- static/attached 倾向稳定，先记 candidate，不轻易改主字段
- dynamic 可在重复证据下确认写回
- 当观测与当前值一致时，会撤销对应冲突候选（revoke）

## 4. 动态生命周期更新

`_update_dynamic_lifecycle(frame_id)` 在每帧末执行：

- 若动态节点本帧未匹配：`life_value -= 1`
- `life_value > 0`: 状态为 `inactive`
- `life_value <= 0`: 状态为 `disappeared`，记录 `disappeared_frame`

若本帧匹配到动态节点：

- 轨迹追加 `(frame_id, box, center)`
- `state=active`
- `life_value` 增长（上限受 `DynamicConfig.max_life_value` 控制）

## 5. 边处理流程（edge_process）

边处理在 `_edge_process()` 中完成。

### 5.1 端点解析

优先使用 `relation.idx` 作为目标端点；若缺失再回退 `object_tag` 映射。

### 5.2 关系文本与结构化字段

每条边同时保存：

- `predicate`
- `source_label`
- `target_label`
- `describe`

其中 `describe` 通过 `_build_edge_describe()` 重建，避免纯字符串拼接导致的抖动。

### 5.3 动作判定（duplicate/conflict/new）

判定流程：

- 若 LLM 可用，先调用 `decide_edge_action`
- 否则回退规则：
  - 同端点同 describe -> duplicate
  - 同端点不同 describe -> conflict（旧边 `invalid_at=frame_id`）
  - 其余 -> new

所有动作都会记录事件到 `graph.edge_action_events`。

## 6. 标签变更后的级联更新

`_rewrite_edges_for_node_changes(frame_id)` 会监听本帧节点标签变更，并更新相关边：

- 若变更节点是边起点，更新 `source_label`
- 若是边终点，更新 `target_label`
- 再调用 `_build_edge_describe` 重建 `describe`

这避免了脆弱的字符串替换问题。

## 7. attached owner 判定与更新

owner 逻辑在 `_resolve_attached_owner()`：

1. 先做谓词门控（owner-like 才继续）
2. 可选 LLM owner 判定（`decide_attached_owner`）
3. 规则更新：
   - 无历史 owner -> assign
   - 与当前 owner 一致 -> keep
   - 冲突 -> 记入 `owner_candidates`
   - 同一候选重复出现达到阈值 -> switch

所有行为写入 `graph.owner_decision_events`，便于追溯。

## 8. LLM 判定机制与回退

LLM 由 `LLMDecider` 提供统一接口，全部要求结构化 JSON 输出。

判定点：

- `decide_frame_dedupe`
- `decide_node_match`
- `decide_edge_action`
- `decide_attached_owner`

可用性判断：

- `LLMConfig.enabled == True`
- `api_key` 非空

任一请求失败或解析失败，返回 `None`，上层自动回退规则逻辑。

## 9. Neo4j 持久化

`Neo4jStorage` 负责：

- `ensure_schema`: 建约束和索引
- `upsert_nodes`: MERGE 节点
- `upsert_edges`: MERGE 边（含 predicate/source_label/target_label/valid_at/invalid_at）

注意：业务判定不在存储层完成，存储层只负责写入。

## 10. 验收报告结构

`build_design_acceptance_report()` 输出：

- `summary`: 总体计数
- `graph_stats`: 图规模、关系动作统计、owner 事件统计
- `samples`: 边动作和 owner 决策样例
- `requirement_checks`: 需求条目逐项 pass/fail
- `overall`: 通过率

## 11. 调试建议

1. 先跑不带 LLM/Neo4j 的纯规则模式，确认流程正确。
2. 再开启 LLM，观察 `samples` 中动作原因是否符合预期。
3. 最后开启 Neo4j，检查图数据库写入的一致性。

## 12. 当前实现边界

- 已实现完整离线主流程与可解释日志。
- 已实现 LLM 可插拔与规则回退。
- 在线 API 尚未实现（当前为离线批处理系统）。
